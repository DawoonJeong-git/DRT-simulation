# 1. 시스템 개요 및 기술 기반

본 장은 본 시스템이 무엇으로 구성되어 있으며, 어떤 원리로 동작하는지를 설명한다.  
이 문서는 파일 나열이 아니라 **“시스템이 어떻게 계산되는지(로직)”**를 이해하는 것이 목적이다.

---

## 1.1 한 줄 정의

본 시스템은 **Segment 기반 운행 데이터를 시간 모델로 해석하여 지도 위에 시각화하는 웹 도구**이다.

---

## 1.2 저장소 범위(포함/미포함)

### 포함
- Frontend: React + MapLibre + deck.gl 기반 시각화
- Backend: Flask 기반 API 서버 (LIVE/REPLAY 모두 Segment 포맷 제공)
- Segment 기반 데이터 파이프라인
- 정적 서비스 범위 데이터: `public/ODD/` 

### 미포함(인수인계/배포 시 별도 필요)
- 운영용 DB 계정/접속 정보
- 지도 타일/스타일 서비스(API Key, Style URL)
- 배포 플랫폼(Render/Vercel 등) 계정 설정

---

## 1.3 기술 스택(역할 중심)

| 영역 | 기술 | 역할 |
|------|------|------|
| Backend API | Python, Flask | Segment 생성 및 제공 |
| Data Source | MySQL (hdl.route, hdl.operation, hdl.reservation_request) | 운행 데이터 저장 |
| Frontend | React | 상태 관리 및 UI |
| Build Tool | Vite | Frontend 번들링 |
| 렌더링 | deck.gl | 차량/경로/상태점 레이어 렌더링 |
| 배경 지도 | MapLibre | 지도 베이스 + 커버리지 레이어 |
| Backend 배포 | Render | API 서버 호스팅 |
| Frontend 배포 | Vercel | 정적 빌드 배포 |

중요한 점:  
> 기술은 구현 수단이며, 시스템의 본질은 “시간 기반 Segment 해석 엔진”이다.

---

## 1.4 실행 환경과 실행 모드(개념 구분)

본 시스템은 **환경(Environment)**과 **모드(Mode)**를 분리한다.
### 실행 환경(Environment)
- OFFLINE: 로컬 Frontend + 로컬 Backend (개발/디버깅)
- ONLINE: 배포 Frontend + 배포 Backend (시연/운영)

환경 전환은 **코드 수정이 아니라 설정(configuration)** 문제다.

### 실행 모드(Mode)
- LIVE: 주기적으로 갱신되는 운행 데이터를 시각화
- REPLAY: 특정 날짜의 운행 데이터를 시간 축으로 재생

LIVE/REPLAY는 **사용 방식의 차이**이며, **데이터 구조는 동일**하다.

---

## 1.5 이 시스템이 계산하는 것

이 시스템은 반복적으로 다음을 계산한다.

> 특정 시각 t에서, 각 차량이 어디에 있는가?

이를 위해 다음을 수행한다.
1. Segment 데이터를 수신한다.
2. 시간 모델을 적용한다.
3. 차량 상태(운행 여부/현재 위치/진행 경로)를 계산한다.
4. 레이어를 생성하고 지도에 렌더링한다.

이 시스템은 예측/배차/운영 제어를 하지 않는다.  
오직 **시각화 계산**만 수행한다. 

---

## 1.6 전체 로직 흐름(End-to-End)

현재 구현 기준 로직 파이프라인은 다음과 같다.

1. DB/JSON에서 route 데이터 조회(Backend)
2. Segment 생성(Backend)
3. API로 segments 배열 제공(Backend)
4. segments 정규화/필터링(Frontend)
5. segments → routeData 변환(Frontend)
6. currentTime 계산(Frontend)
7. 차량 위치/진행 경로 계산(Frontend)
8. deck.gl 레이어 생성 및 렌더링(Frontend)

LIVE와 REPLAY는 위 파이프라인을 공유한다.

---

## 1.7 핵심 설계 원칙

### 원칙 1 — Segment 중심 계약
Segment는 Backend와 Frontend 사이의 유일한 계약이다.  
DB 스키마/로딩 방식은 바뀔 수 있으나, Segment 계약이 유지되면 Frontend는 안정적으로 동작한다.

### 원칙 2 — Frontend 주도 시간 해석
시간 모델은 Frontend가 해석한다.
- Backend는 “재생 시간”을 모르며, 데이터를 제공할 뿐이다.
- REPLAY의 빠른 시간 이동/속도 변경을 가능하게 한다.

### 원칙 3 — 계산과 표현의 분리
차량 위치/진행 경로 계산은 Frontend에서 끝낸다.  
deck.gl은 계산 결과를 표현하는 렌더링 계층이다.

---

## 1.8 절대 불변 조건(Invariants)

다음 조건은 반드시 유지되어야 한다.
1. `originMs < destMs`
2. `polyline`은 최소 2개 이상의 `[lon, lat]` 좌표를 가진다.
3. 모든 시간 값은 epoch millisecond(정수)이다.
4. `vehicleID + operationID` 조합은 운행 단위를 식별한다.
5. Segment 필드명/의미는 변경하지 않는다.

이 조건이 깨지면 시스템은 정상 동작하지 않는다.

---

## 1.9 환경 전환 시 1차 점검 체크리스트

OFFLINE ↔ ONLINE 전환 또는 배포 시, 아래를 먼저 확인한다.

- Frontend의 API base URL이 올바른 Backend를 가리키는가?
- MapLibre 스타일 URL / 타일 API Key가 유효한가?
- Backend 환경변수(DB_*, ALLOWED_ORIGINS 등)가 설정되어 있는가?
- LIVE/REPLAY API가 동일한 Segment 배열 구조로 응답하는가?

---

## 1.10 시스템 경계(Scope Boundary)

본 시스템은 다음을 의도적으로 포함하지 않는다. 

- 사용자 인증/권한 관리
- 실시간 배차 제어 또는 최적화
- 운영 모니터링 알림
- 예측 분석

본 시스템은 운영 플랫폼이 아니라 **시각화/분석 보조 도구**이다. 





# 2. Backend 로직

본 장은 Backend가 원천 운행 데이터를 **Segment 계약 구조**로 변환하여 제공하는 방식을 설명한다.

Frontend는 DB를 알지 못하며, 오직 Backend가 제공하는 **segments 배열(=Segment 계약)**만 신뢰한다.  
따라서 Backend의 핵심 책임은 다음 한 줄로 요약된다.

> DB/JSON 데이터를 안정적인 Segment 포맷으로 변환해 API로 제공한다.

---

## 2.1 Backend의 책임과 경계

### Backend가 하는 일
1. DB 또는 JSON에서 운행 데이터 조회
2. 조회 결과를 Segment 단위로 변환(build_segments)
3. 예약 데이터(reservation_request)를 이벤트(events)로 집계
4. LIVE/REPLAY API로 동일한 Segment 배열 반환

### Backend가 하지 않는 일
- 재생 시간 제어(elapsedTime)
- 차량 위치 계산(보간)
- 렌더링 판단/레이어 생성
- UI 상태 관리

시간 해석과 렌더링은 Frontend 책임이다. 

---

## 2.2 데이터 원천과 조회 규칙

Backend는 기본적으로 다음 테이블을 사용한다.

- `hdl.route`
- `hdl.operation` (JOIN)
- `hdl.reservation_request` (dispatchID 기반 조회)

### 2.2.1 route + operation JOIN에서의 alias/우선순위

Backend는 DB 필드를 Frontend 계약에 맞추기 위해 다음을 유지한다.

- `destArrivalTime` → `destDeptTime` (alias 유지)
- `operation.vehicleID` → `op_vehicleID` (alias)
- `VehicleType` → `vehicleType` (alias)

vehicleID는 다음 우선순위를 가진다.

1. `route.vehicleID`
2. (없으면) `op_vehicleID`

이 규칙은 운행 식별 안정성을 위한 변경 금지 규칙이다.

### 2.2.2 일 단위 조회 범위(날짜 필터)

일 단위 조회는 `originDeptTime`(숫자형 YYYYMMDDHHMM 형태)을 기준으로 날짜 범위를 자른다.

- start: `{YYYYMMDD}0000`
- end: `{YYYYMMDD}2359`

이 규칙은 REPLAY “해당 날짜 데이터”의 정의와 직접 연결된다.

---

## 2.3 Segment 단위 정의

Segment는 “routeSeq 기반 이동 단위”이다.

- 하나의 운행(`operationID`)은 여러 `routeSeq`를 포함할 수 있다.
- 각 `routeSeq`가 하나의 Segment로 변환된다.
- Segment 식별자는 일반적으로 `{routeID}:{routeSeq}` 형태(`segment_id`)로 구성된다. 

Frontend는 Segment를 시간 계산의 최소 단위로 사용한다.

---

## 2.4 Segment 생성 로직(build_segments)

Segment 생성은 “기본 Segment 생성 → 이벤트 집계”의 2단계로 구성된다.

### 2.4.1 1단계: 기본 Segment 생성

routes(각 route row)를 순회하며 Segment를 만든다.

필수 핵심 필드(Backend → Frontend 계약):
- `vehicleID`
- `operationID`
- `originMs` (epoch ms)
- `destMs` (epoch ms)
- `polyline` (좌표 배열, [lon, lat])

추가로 구현상 함께 포함되는 대표 필드:
- `routeID`, `routeSeq`, `segment_id`
- `vehicleType`, `routeInfo`
- `originStationID`, `destStationID`
- `originDeptTime`, `destDeptTime`
- `dispatchIDs`
- `events`

### 2.4.2 시간 변환 규칙(originMs/destMs)

Backend는 모든 시간 값을 **epoch millisecond 정수**로 제공한다.

- 문자열 시간 포맷은 API 계약에 포함되지 않는다.
- Frontend는 `originMs/destMs`를 기준으로만 시간 계산을 수행한다.

불변조건:
- `originMs < destMs`는 반드시 성립해야 한다.

### 2.4.3 polyline 생성 규칙

polyline은 다음 규칙을 따른다.

- 좌표 순서: `[lon, lat]`
- 최소 2개 좌표 이상
- 좌표 순서는 이동 순서와 일치해야 한다.

polyline이 깨지면 Frontend의 위치 보간이 불가능해진다.

---

## 2.5 reservation 이벤트 집계(events)

Backend는 예약 데이터(reservation_request)를 dispatchID로 조회하여 Segment.events에 반영한다.

### 2.5.1 events 구조(고정 형태, 변경 금지)

events는 origin/dest에 대해 board/alight 형태를 가진다.
Frontend가 이 구조를 전제로 하므로, 필드명/중첩 구조 변경은 금지한다.

### 2.5.2 집계 로직 개요

- `dispatchIDs`로 reservation_request를 조회한다.
- reservation의 `dropoffStationID` 기준으로 “어느 Segment의 dest에 누적할지”를 찾는다.
- 해당 Segment의 `events.dest.alight`에 누적한다.

### 2.5.3 집계 실패 시 동작(중요)

reservation 연결 실패는 **치명적 실패가 아니다**.

- Segment는 생성되어야 한다(데이터 제공은 지속)
- events는 기본값(0)을 유지할 수 있다

즉, “Segment 생성”은 reservation에 의존하지 않는다.

---

## 2.6 API 제공 구조(LIVE/REPLAY)

Backend는 LIVE/REPLAY 모두 **동일한 segments 배열 구조**를 제공한다.

- LIVE: 최신 데이터 스냅샷을 주기적으로 제공 (Frontend가 polling)
- REPLAY: 특정 날짜 데이터 묶음을 1회 제공 (재생은 Frontend가 수행)

LIVE/REPLAY 차이는 “제공 방식”이며, 계약 구조는 동일하다.

---

## 2.7 Backend 동작 파라미터(환경변수 요약)

Backend는 실행 환경 전환을 코드 분기 대신 환경변수로 처리한다. 

- DB 접속: `DB_USER`, `DB_PASS`, `DB_HOST`, `DB_PORT`, `DB_NAME`
- 서버: `HOST`, `PORT`
- CORS: `ALLOWED_ORIGINS`
- LIVE 갱신: `POLL_INTERVAL`
- REPLAY 캐시(사용 시): `REPLAY_CACHE_TTL`

실제 값은 문서에 기록하지 않고 “키 이름과 역할”만 유지한다.

---

## 2.8 Backend 불변조건(Invariants)

다음 조건은 Backend에서 반드시 유지되어야 한다.

1. `originMs < destMs`
2. `polyline` 최소 2개 좌표
3. 시간 값은 epoch millisecond 정수
4. `vehicleID + operationID`는 운행 단위 식별자
5. events 구조(origin/dest board/alight) 변경 금지
6. Segment 필드명/의미 변경 금지

이 조건이 깨지면 Frontend는 정상 동작하지 않는다.







# 3. Frontend 로직

Frontend는 이 시스템의 핵심 계산 엔진이다.

Backend는 Segment(segments 배열)를 제공할 뿐이며,
Frontend는 이를 기반으로 **시간 해석 → 차량 상태 계산 → 레이어 생성**을 수행한다.

---

## 3.1 Frontend 역할 분리(Architecture)

Frontend 로직은 세 계층으로 분리된다.

1) Orchestrator: `CombinedView.jsx`  
- LIVE/REPLAY 데이터 로딩
- 시간 상태(baseDateMs, elapsedTime) 관리
- 필터링(areaMode) 및 레이어 생성 호출
- deckLayers 최종 상태 유지

2) Adapter: `src/deck/adapters/segmentsAdapter.js`  
- segments → routeData 변환(그룹핑/정렬/검증)

3) Renderer: `src/deck/VehicleLayer.jsx`  
- routeData + currentTime → 차량 상태/위치 계산
- deck.gl layers 생성

---

## 3.2 상태 모델(State Model)

### 3.2.1 핵심 상태 변수

- `isLive`: LIVE/REPLAY 모드 플래그
- `baseDateMs`: 선택한 날짜의 자정(epoch ms)
- `elapsedTime`: 재생으로 증가하는 경과 시간(초)
- `isPlaying`: 재생 중 여부
- `speed`: 재생 속도 배율
- `areaMode`: `both | accessible | underserved`
- `routeData`: segments를 운행 단위로 변환한 렌더링 입력
- `deckLayers`: deck.gl에 전달되는 최종 레이어 배열

### 3.2.2 상태 전이(Trigger → 재계산)

| Trigger | 변경되는 상태 | 후속 계산 |
|---|---|---|
| LIVE polling 응답 | segments/routeData | normalize → routeData 재생성 → 레이어 재계산 |
| REPLAY 날짜 변경 | baseDateMs + segments/routeData | routeData 재생성 + elapsedTime 초기화 |
| 재생 시작/정지 | isPlaying | elapsedTime 증가/정지 |
| 속도 변경 | speed | elapsedTime 증가율 변경 |
| 서비스 모드 변경 | areaMode | filteredRouteData 변경 → 레이어 재계산 |

---

## 3.3 데이터 파이프라인(Data Pipeline)

Frontend에서의 데이터 흐름은 다음과 같다.

### 3.3.1 API 로딩(LIVE/REPLAY)

- LIVE: `/api/segments`를 일정 주기로 호출(polling) 
- REPLAY: `/api/replay?date=YYYY-MM-DD`를 날짜 변경 시 1회 호출

두 모드는 **segments 배열 구조가 동일**하며, 차이는 “갱신 방식”이다.

### 3.3.2 segments 정규화(normalize) 및 일 단위 필터

`CombinedView.jsx`는 수신한 segments에 대해:

- `originMs/destMs` 보정(누락 시 시간 필드에서 변환)
- `baseDateMs` 기준 하루 범위에 걸치는 segment만 유지
- 필수 필드/폴리라인 유효성 1차 검증

출력: “재생 가능한 segments”

### 3.3.3 segments → routeData 변환(Adapter)

`segmentsAdapter.js`는 다음을 수행한다.

1. `vehicleID + operationID` 기준 그룹핑(운행 단위)
2. 시간 순 정렬(originMs/destMs)
3. polyline 최소 조건(2점 이상) 검증
4. 렌더링에 적합한 run 구조(routeData) 생성

출력: `routeData`

### 3.3.4 areaMode 필터(서비스 모드)

`areaMode`는 `vehicleTypeToAreaMode(vehicleType)` 매핑으로 결정된다. 
- `areaMode` 변경 → `filteredRouteData` 변경 → 레이어 재계산

---

## 3.4 시간 모델(Time Model)

Frontend는 다음 시간 모델을 사용한다.

- `baseDateMs`: 날짜 자정(epoch ms)
- `elapsedTime`: 경과 시간(초)

실제 비교에 쓰이는 시간은 다음과 같다.

- `currentTimeMs = baseDateMs + elapsedTime * 1000`

즉, Segment의 `originMs/destMs`(epoch ms)와 동일 단위로 맞춘 뒤 비교한다.

### 3.4.1 LIVE 모드 해석

LIVE는 “현재를 보여주는 모드”이지만,
Frontend 내부 계산은 여전히 `baseDateMs + elapsedTime` 구조를 유지한다.

핵심은 다음 두 흐름이 동시에 존재한다는 점이다.

1) elapsedTime(재생 시계)은 계속 증가  
2) routeData는 polling으로 교체

이 둘은 서로 독립이며, polling이 와도 시간 모델 자체는 바뀌지 않는다.

### 3.4.2 REPLAY 모드 해석

REPLAY는 “데이터는 고정”이고 “시간만 움직인다”.

- 데이터는 날짜 선택 시 1회 로딩
- 이후 재생/속도 변경은 Frontend에서만 처리
- Backend는 재생 시간에 관여하지 않는다

---

## 3.5 차량 상태 계산(Vehicle State Computation)

`VehicleLayer.jsx`는 `filteredRouteData`와 `currentTimeMs`를 입력으로 받아
각 차량의 “현재 상태”를 계산한다.

### 3.5.1 운행 중 여부 판단

각 Segment에 대해 다음 조건을 평가한다.

- 운행 중: `originMs ≤ currentTimeMs ≤ destMs`

운행 중이 아니면(출발 전/도착 후) 기본적으로 위치를 표시하지 않거나,
진행 구간 표시를 생략한다(정책에 따라 다름).

### 3.5.2 진행률 계산(시간 비율)

시간 기반 진행률:

- `t_ratio = (currentTimeMs - originMs) / (destMs - originMs)`  
- 범위: 0~1 (클램프 필요)

※ t_ratio는 “목표 지점”을 찾기 위한 기준이며,
실제 위치는 polyline 거리 기반으로 계산한다.

### 3.5.3 polyline 거리 기반 위치 보간(핵심)

현재 위치 계산은 다음 절차를 따른다.

1. polyline의 구간별 거리(점-점) 계산
2. 누적 거리 배열(cumulative distance) 생성
3. 목표 거리: `targetDist = t_ratio * totalDist`
4. targetDist가 속한 구간을 찾고 선형 보간
5. 결과: `currentPosition = [lon, lat]`

이 방식은 polyline이 굽어도 자연스럽게 진행 위치를 제공한다.

### 3.5.4 경로 분리(전체 vs 진행 구간)

- 전체 경로: polyline 전체를 PathLayer로 렌더링
- 진행 구간: currentTimeMs까지의 부분 경로를 별도 PathLayer로 렌더링(강조)

---

## 3.6 레이어 생성(deck.gl Layers)

`VehicleLayer.jsx`는 보통 다음 레이어를 생성한다.

1. PathLayer (전체 경로)
2. PathLayer (진행 구간)
3. IconLayer (차량 아이콘)
4. ScatterplotLayer (상태점)

> deck.gl은 계산 결과를 “표현”하는 계층이다.
> 시간/상태 계산은 이미 VehicleLayer 수준에서 완료되어야 한다.

---

## 3.7 지도 계층(MapLibre)과의 역할 분리

지도는 “배경(MapLibre)”과 “객체(deck.gl)”로 분리되어 있다. 

- MapLibre: 배경 지도 + 커버리지(정류장/링크) 레이어
- deck.gl: 차량/경로/상태점 + pick/hover 등 인터랙션 객체

MapLibre의 인터랙션을 과도하게 활성화하면
deck.gl 인터랙션과 충돌할 수 있다.

---

## 3.7.1 초기 뷰(카메라) 결정 로직

“처음 켰을 때 어디를 보여주는가”는 `CombinedView.jsx`의 책임이다. 

- `useStationCoords()`로 정류장 좌표를 로드
- 좌표로 bounds 계산
- `fitViewStateToBounds()`로 초기 ViewState 설정

---

## 3.7.2 레이어 ID 중복 방지(dedupe)

`deckLayers`는 레이어 `id`가 중복되면 깜빡임/표시 이상이 발생할 수 있다.

따라서 `dedupeLayersById()`를 적용해 중복 id를 제거하는 것이 안전하다. 

---

## 3.8 렌더링 재계산 트리거 구조

Frontend는 특정 상태 변화가 발생할 때만 deckLayers를 재생성한다.

다음 조건 중 하나가 변경되면 재계산이 발생한다.

- elapsedTime 변경
- routeData 변경
- areaMode 변경
- LIVE polling 응답 수신

재계산 구조는 다음과 같다.

routeData + currentTime → vehicle state → deckLayers

즉, 렌더링은 상태 변화의 결과이지 독립적으로 발생하지 않는다.

---

## 3.9 LIVE 모드에서의 시간과 데이터 갱신 관계

LIVE 모드에서는 두 흐름이 동시에 존재한다.

1. elapsedTime은 지속적으로 증가한다.
2. routeData는 60초마다 교체된다.

이 둘은 서로 독립적이다.

routeData가 교체되더라도 시간 모델(currentTime 계산 방식)은 유지된다.

이 구조 덕분에 LIVE 모드에서도 시간 흐름은 일관되게 유지된다.

---

## 3.10 시간 해석 예외 케이스

Segment에 대해 다음 경우가 존재한다.

1. currentTime < originMs  
   → 아직 출발 전  
   → 차량 표시하지 않음

2. currentTime > destMs  
   → 도착 후  
   → 진행 구간 표시하지 않음

3. polyline 길이 < 2  
   → 위치 보간 불가  
   → 해당 Segment 제외

4. routeData가 비어 있음  
   → deckLayers 생성 생략

이 예외 처리는 위치 계산 안정성을 보장한다.

---

## 3.11 Frontend 불변 조건(Invariants)

다음 조건은 Frontend 로직 안정성을 위해 반드시 유지되어야 한다.

1. routeData는 시간 순 정렬되어야 한다.
2. polyline은 [lon, lat] 순서 및 최소 2점 조건을 만족해야 한다.
3. elapsedTime은 음수가 될 수 없다.
4. 상태 변경은 항상 재계산을 트리거해야 한다.
5. MapLibre(배경)와 deck.gl(객체/인터랙션)의 역할 분리를 유지해야 한다.





# 4. 기능별 로직 체인 및 동작 흐름(실전/디버깅 중심)

본 장은 시스템의 주요 기능이 실제로 어떤 순서로 동작하는지를 정리한다.

핵심 관점은 다음이다.

- 이벤트(Trigger)
- 상태 변화(State change)
- 계산(Computation)
- 렌더링(Render)

즉, “이벤트 → 상태 변화 → 계산 → 렌더” 체인으로 문제를 추적한다.

---

## 4.1 LIVE 모드 동작 흐름

LIVE 모드는 “데이터는 갱신되고, 시간은 계속 흐르는 상태”이다.

### 4.1.1 초기 진입(첫 로드)

1) isLive = true  
2) `/api/segments` 호출  
3) segments 수신  
4) normalize + 일 단위 필터  
5) segments → routeData 변환  
6) areaMode 필터 적용 → filteredRouteData  
7) getVehicleLayers(filteredRouteData, currentTime)  
8) deckLayers 렌더링

### 4.1.2 polling 갱신(예: 60초)

1) `/api/segments` 재호출  
2) segments 수신  
3) normalize + 일 단위 필터  
4) routeData 교체  
5) elapsedTime은 유지(시간 모델은 유지)  
6) vehicle state 재계산  
7) deckLayers 재생성

주의:
- LIVE에서 “routeData 갱신”과 “elapsedTime 증가”는 서로 독립이다.

---

## 4.2 REPLAY 모드 동작 흐름

REPLAY는 “데이터는 고정, 시간만 사용자가 움직임”이다.

### 4.2.1 날짜 선택(데이터 로딩)

1) baseDateMs 설정(해당 날짜 자정)  
2) `/api/replay?date=YYYY-MM-DD` 호출  
3) segments 수신  
4) normalize + 일 단위 필터  
5) segments → routeData 변환  
6) elapsedTime = 0 초기화  
7) deckLayers 생성

### 4.2.2 재생 시작(시간 흐름)

1) isPlaying = true  
2) elapsedTime 증가(속도 speed 적용)  
3) currentTimeMs = baseDateMs + elapsedTime*1000  
4) vehicle state 재계산  
5) deckLayers 재생성

Backend는 재생에 관여하지 않는다. (데이터 제공만 수행)

---

## 4.3 서비스 모드(areaMode) 변경 흐름

areaMode는 차량 필터와 커버리지 레이어에 동시에 적용된다.

1) areaMode 변경(both / accessible / underserved)  
2) vehicleTypeToAreaMode 매핑 적용  
3) filteredRouteData 재생성  
4) deckLayers 재생성(차량/경로 변경)  
5) MapLibre 커버리지 레이어도 mode 동기화(표현 변경)

---

## 4.4 coverageVisible 토글 흐름(커버리지 ON/OFF)

coverageVisible은 “커버리지 레이어”에만 영향이 있다.

1) coverageVisible 변경  
2) MapLayers.jsx에서 정류장/링크 source/layer 표시/숨김  
3) deckLayers(차량 레이어)에는 영향 없음

즉, 차량이 안 보이는 문제는 coverageVisible과 무관한 경우가 대부분이다.

---

## 4.5 초기 화면 범위(카메라 fit) 동작 흐름

“처음 켰을 때 지도 어디를 보여주는가”는 CombinedView의 책임이다. 

1) useStationCoords로 정류장 좌표 로드  
2) 좌표로 bounds 계산  
3) fitViewStateToBounds로 초기 viewState 설정  
4) MapLibre + deck.gl 카메라 동기화

초기 뷰가 이상하면(너무 멀거나 엉뚱한 곳) stationCoords 로딩/키 매칭을 먼저 의심한다.

---

## 4.6 “차량이 화면에 나타나는” 필수 조건

차량이 정상적으로 보이기 위한 조건은 다음과 같다.

1) routeData(또는 filteredRouteData)가 비어있지 않다  
2) currentTimeMs가 해당 Segment의 시간 범위에 들어간다  
   - originMs ≤ currentTimeMs ≤ destMs  
3) polyline 길이 ≥ 2 (좌표 2점 이상)  
4) areaMode 필터를 통과한다(vehicleType 매핑 포함)  
5) (선택) 해당 차량 아이콘 리소스가 로드 가능하다(public 경로/URL 정상)

이 중 하나라도 실패하면 차량은 표시되지 않을 수 있다.

---

## 4.7 인수인계/배포 후 코어 검증 시나리오(필수 4개)

아래 4개는 “정상 동작 확인”의 최소 세트다.

1) LIVE 모드에서 차량이 주기적으로 갱신되는가?  
2) REPLAY 모드에서 날짜 선택 시 차량이 나타나는가?  
3) areaMode를 바꾸면 차량/커버리지 표시가 달라지는가?  
4) coverageVisible 토글이 정류장/링크 표시를 제어하는가?

---

## 4.8 차량이 안 뜰 때(최우선 디버깅 루트)

“차량이 안 뜬다”는 대부분 아래 4가지 중 하나다.

### Step 1 — API 응답(segments) 확인
- LIVE: `/api/segments`가 segments 배열을 주는가?
- REPLAY: `/api/replay?date=...`가 segments 배열을 주는가?
- segments가 비어 있으면 Backend/DB/날짜 필터 문제 가능성이 높다.

### Step 2 — routeData 생성 여부 확인
- normalize 이후 필터가 과도하게 적용되어 routeData가 비는지 확인
- vehicleID/operationID가 누락되면 그룹핑이 실패할 수 있다.

### Step 3 — 시간 범위 확인
- currentTimeMs가 originMs~destMs에 들어가는지 확인
- baseDateMs(자정) 기준이 잘못되면 “하루 전체가 어긋나” 차량이 안 뜰 수 있다.

### Step 4 — polyline 검증
- polyline이 2점 미만이면 제외된다.
- [lat, lon] 순서가 뒤집히면 화면 밖으로 날아갈 수 있다.

### Step 5 — areaMode 매핑 확인
- vehicleTypeToAreaMode 결과가 기대와 다르면 필터에서 전부 사라질 수 있다.

---

## 4.9 경로가 안 그려질 때(차량은 보이는데 Path가 없음)

1) polyline 존재/길이 확인(2점 이상)  
2) polyline 좌표 순서([lon, lat]) 확인  
3) “전체 경로 레이어”와 “진행 구간 레이어”가 모두 생성되는지 확인  
4) 레이어 id 중복으로 덮어쓰는지 확인(dedupe 적용 여부)

---

## 4.10 시간이 이상할 때(재생/현재 위치가 틀어짐)

1) baseDateMs가 선택한 날짜의 자정인지 확인  
2) elapsedTime 단위 확인(초)  
3) currentTimeMs 계산이 ms 단위로 변환되어 비교되는지 확인  
4) originMs/destMs가 epoch ms인지 확인(숫자 단위 불일치 여부)

---

## 4.11 레이어가 깜빡이거나 중복될 때

대부분 “레이어 id 중복” 또는 “불필요한 재생성” 문제다.

1) deckLayers 내 id 중복 여부 확인  
2) dedupeLayersById 적용 여부 확인  
3) useEffect 의존성 배열이 과도하게 넓어 불필요한 재계산이 발생하는지 확인  
4) polling 갱신 시 routeData 교체 타이밍에 의도치 않은 초기화가 없는지 확인

---

## 4.12 환경 전환/배포 후 바로 점검할 것(증상 기반)

환경 전환은 설정 문제이며, 아래는 1차 점검 항목이다.

- API base URL이 올바른 Backend를 가리키는가?
- CORS(ALLOWED_ORIGINS) 설정이 맞는가?
- MapLibre style URL / 타일 API Key가 유효한가?
- LIVE/REPLAY API가 동일한 Segment 구조를 반환하는가?

---

## 4.13 성능/대용량 데이터에서 흔한 증상과 원인

현재 구조는 polling 기반 갱신이며(의도적 단순화), 대용량 Segment에서 성능 이슈가 발생할 수 있다.

자주 보이는 증상:
- 화면 렌더링 지연
- polling 시 프레임 드랍/깜빡임
- hover/pick 반응 저하

점검 포인트:
- segments 수(차량×시간 구간)가 과도한지
- polyline 점 수가 과도한지
- 불필요한 레이어 재생성이 있는지(3.8 트리거 확인)

---

## 4.14 기능별 로직 요약표

| 기능 | Trigger | 계산 변화 | 렌더 변화 |
|---|---|---|---|
| LIVE 갱신 | polling | routeData 교체 | 레이어 재생성 |
| REPLAY 재생 | elapsedTime 증가 | vehicle 위치 변경 | 레이어 재생성 |
| 모드 변경 | areaMode 변경 | 필터링 | 레이어 재생성 |
| coverage 토글 | coverageVisible 변경 | 없음 | MapLibre 레이어 변경 |
| 초기 뷰 fit | stationCoords 로드 | bounds 계산 | viewState 설정 |

---

## 4.15 정리

이 시스템의 문제 추적은 항상 다음 체인으로 한다.

**이벤트 → 상태 변화 → 계산 → 렌더링**

routeData + currentTime → vehicle state → deckLayers

이 체인을 기준으로 보면,
대부분의 “차량/시간/표시” 문제를 빠르게 좁힐 수 있다.






# 5. 설계 결정 및 제약 사항

본 장은 시스템이 현재 구조로 설계된 이유와, 의도적으로 유지하고 있는 제약을 명확히 기술한다.

이 장의 목적은 다음과 같다.

- 다음 개발자가 구조를 불필요하게 복잡하게 만들지 않도록 한다.
- 변경 시 어떤 부분이 위험한지 명확히 한다.
- 현재 구조의 한계를 인지한 상태에서 확장하도록 한다.

---

## 5.1 Segment 중심 설계(Segment-Centric Design)

### 설계 의도

본 시스템은 모든 시각화를 Segment 단위 데이터에 의존하도록 설계되었다.

이 설계의 목적은 다음과 같다.

- DB 스키마 변경이 Frontend에 직접 영향을 주지 않도록 하기 위함
- LIVE / REPLAY를 동일한 렌더링 구조로 유지하기 위함
- Backend 내부 구현 변경을 Frontend와 분리하기 위함

### 결과

- Segment는 Backend와 Frontend 사이의 유일한 계약이다.
- Segment 구조가 깨지면 Frontend는 정상 동작하지 않는다.
- 반대로 DB 구조 변경은 Backend 내부에서 흡수 가능하다.

---

## 5.2 Frontend 주도 시간 모델(Frontend-Driven Time Model)

### 설계 의도

재생 시간 제어는 전적으로 Frontend의 책임이다.

이 설계의 목적은 다음과 같다.

- Backend 단순화
- REPLAY 모드에서 빠른 시간 이동 및 속도 변경 가능
- LIVE / REPLAY 렌더링 로직 통합

### 제약

- 시간 계산 오류는 모두 Frontend 로직 문제이다.
- elapsedTime 오작동 시 전체 렌더링이 왜곡된다.
- Backend는 “현재 시각” 개념을 알지 못한다.

---

## 5.3 MapLibre와 deck.gl 역할 분리

### 설계 의도

지도는 MapLibre, 오브젝트 렌더링은 deck.gl로 분리하였다.

목적:

- 배경 지도와 객체 인터랙션 분리
- 성능 최적화
- 렌더링 계층 역할 명확화

### 주의 사항

- MapLibre 인터랙션을 과도하게 활성화하면 deck.gl과 충돌 가능
- 카메라 동기화(viewState 공유)는 항상 일관되게 유지해야 한다

---

## 5.4 vehicleType 기반 서비스 구분

### 설계 의도

서비스 모드는 vehicleType 기반으로 매핑한다. 

이 설계의 목적:

- DB 구조 변경 없이 서비스 구분 가능
- 필터 로직 단순화

### 제약

- vehicleType 정의 변경 시 Frontend 수정 필요
- vehicleTypeToAreaMode 매핑 함수 변경은 영향 범위가 넓다

---

## 5.5 LIVE polling 구조 유지 이유

### 설계 의도

LIVE 모드는 polling 기반 데이터 갱신 구조를 유지한다.

목적:

- 구현 단순성 유지
- 스트리밍 서버(WebSocket 등) 불필요

### 한계

- 초 단위 실시간 반영은 불가
- 대용량 데이터에서 polling 시 프레임 저하 가능
- 데이터 갱신과 시간 흐름이 동시에 발생할 경우 디버깅 난이도 상승

---

## 5.6 Intentional Omissions (의도적 제외 항목)

본 시스템은 다음을 의도적으로 포함하지 않았다.

- 사용자 인증
- 권한 관리
- 대규모 상태 관리 라이브러리(Redux 등)
- 자동 배포 파이프라인
- 실시간 배차 제어/최적화

이 시스템은 운영 플랫폼이 아니라 시각화 도구이다.

---

## 5.7 Known Constraints (명시적 기술 부채)

현재 구조는 단순성과 유지보수성을 우선한 설계이다.

따라서 다음 한계를 인지해야 한다. 

1. 대용량 Segment 처리 시 성능 저하 가능
2. polyline 거리 기반 보간은 단순 선형 보간(고급 보간 아님)
3. LIVE polling 기반 구조의 실시간성 한계
4. MapLibre 스타일 URL이 코드에 직접 참조될 수 있음
5. 모든 시간 계산이 Frontend에 집중되어 있음

이 항목들은 “문제가 있어서”가 아니라, 현재 목적 대비 과도한 복잡도를 피하기 위해 유지한 설계이다.

---

## 5.8 변경 시 위험 구간(High-Risk Modification Zones)

다음 변경은 시스템 전반에 영향을 줄 수 있다.

### 매우 위험

- Segment 필드 구조 변경
- originMs/destMs 단위 변경
- vehicleID + operationID 그룹핑 기준 변경
- polyline 좌표 순서 변경([lat, lon]로 뒤집힘 등)

### 중간 위험

- vehicleTypeToAreaMode 매핑 변경
- areaMode 필터 로직 수정
- elapsedTime 계산 방식 변경

### 상대적으로 안전

- 아이콘 이미지 교체
- 경로 색상 변경
- UI 레이아웃 변경

---

## 5.9 확장 시 권장 원칙

확장 또는 리팩토링 시 다음을 유지해야 한다.

4. 상태 변화 → 재계산 트리거 구조를 유지할 것
5. MapLibre와 deck.gl의 역할 분리를 유지할 것

# Appendix B — API Contract & Segment Example

본 부록은 Backend와 Frontend 사이의 핵심 계약인
Segment 구조를 실제 예시(JSON)로 명확히 정의한다.

이 구조가 유지되는 한,
DB 스키마나 내부 로직은 변경 가능하다.

---

## B.1 API 응답 구조

LIVE 및 REPLAY API는 동일한 구조를 반환한다.

{
  "segments": [
    {
      "routeID": "R20240201001",
      "segment_id": "R20240201001:3",
      "operationID": "OP_1002",
      "vehicleID": "CAR_07",
      "vehicleType": "carnivalWheel",
      "originStationID": "S12",
      "destStationID": "S18",
      "originDeptTime": 202402011030,
      "destDeptTime": 202402011045,
      "originMs": 1706754600000,
      "destMs": 1706755500000,
      "polyline": [
        [127.3123, 36.3745],
        [127.3140, 36.3751],
        [127.3189, 36.3788]
      ],
      "dispatchIDs": ["D10023", "D10024"],
      "events": {
        "origin": {
          "board": 1,
          "alight": 0
        },
        "dest": {
          "board": 0,
          "alight": 2
        }
      }
    }
  ]
}

LIVE와 REPLAY의 차이는 데이터 제공 방식이며,
응답 구조는 완전히 동일하다.

---

## B.2 필수 필드 (Frontend 계약 핵심)

다음 필드는 반드시 존재해야 한다.

- vehicleID
- operationID
- originMs
- destMs
- polyline

이 중 하나라도 누락되면
Frontend는 정상 계산을 수행할 수 없다.

---

## B.3 시간 계약 규칙

- 모든 시간 값은 epoch millisecond 정수이다.
- originMs는 destMs보다 반드시 작아야 한다.
- Frontend는 originMs와 destMs만을 기준으로 시간 계산을 수행한다.
- 문자열 시간 포맷은 API 계약에 포함되지 않는다.

---

## B.4 polyline 계약 규칙

- 좌표 순서는 [lon, lat]이다.
- 최소 2개 이상의 좌표를 가져야 한다.
- 이동 순서와 동일해야 한다.

좌표 순서가 [lat, lon]으로 바뀌면
차량 위치 계산이 완전히 틀어질 수 있다.

---

## B.5 events 구조 계약

events는 다음 구조를 유지해야 한다.

{
  "origin": { "board": number, "alight": number },
  "dest":   { "board": number, "alight": number }
}

필드명 또는 계층 구조 변경은 금지한다.

---

## B.6 변경 금지 항목

- 필드명 변경 금지
- 시간 단위 변경 금지
- polyline 좌표 구조 변경 금지
- events 구조 변경 금지

필드 추가는 가능하나,
Frontend가 해당 필드에 강하게 의존하도록 만들지 않는다.

---

## B.7 정리

Segment는 Backend와 Frontend 사이의 유일한 계약이다.

Segment 구조가 유지되는 한,
Frontend는 안정적으로 동작한다.





# Appendix C — Environment & External Dependency Reference

본 부록은 실행 환경, 외부 의존성, 환경변수 구성에 대한 “구조적 정리”이다.

이 장의 목적은 다음과 같다.

- 환경 전환 시 무엇이 바뀌는지 명확히 한다.
- 외부 서비스 의존 지점을 한눈에 정리한다.
- 코드 수정 없이 설정으로 해결해야 할 영역을 분리한다.

---

## C.1 실행 환경 개념 정리

본 시스템은 두 가지 실행 환경을 가진다.

### OFFLINE 환경
- 로컬 Frontend + 로컬 Backend
- 개발/디버깅 목적
- DB 또는 로컬 JSON 사용 가능

### ONLINE 환경
- 배포된 Frontend + 배포된 Backend
- 시연/운영 목적
- 환경변수 기반 DB 및 외부 서비스 설정

중요 원칙:

> ONLINE / OFFLINE 차이는 코드 차이가 아니라 설정 차이이다.

---

## C.2 환경변수 구성 요약

Backend 실행에 필요한 주요 환경변수는 다음과 같다.


### DB 관련

- `DB_USER`
- `DB_PASS`
- `DB_HOST`
- `DB_PORT`
- `DB_NAME`

### 서버 실행 관련

- `HOST`
- `PORT`
- `ALLOWED_ORIGINS`

### LIVE/REPLAY 동작 관련

- `POLL_INTERVAL`
- `REPLAY_CACHE_TTL`

원칙:

- 실제 값은 문서에 기록하지 않는다.
- 키 이름과 역할만 유지한다.
- 운영 환경에서는 환경변수로 주입한다.

---

## C.3 Frontend 환경 연결 지점

Frontend는 다음 설정 지점에서 환경에 의존한다.

### 1) API Base URL

파일: `src/utils/api.js`

- 로컬 개발: `http://localhost:PORT`
- 배포 환경: `https://<render-domain>`

API base URL이 잘못 설정되면
LIVE/REPLAY가 모두 동작하지 않는다.

---

### 2) 지도 스타일 및 타일 서비스

파일: `src/maplibre/MapView.jsx`

- style URL
- 타일 API key

이 값이 개인 계정에 종속되어 있을 수 있다.

운영 시:

- 공용 계정 사용
- 또는 환경변수 기반 주입 권장

---

## C.4 외부 서비스 의존성 목록

본 프로젝트는 다음 외부 서비스에 의존할 수 있다.

1) 지도 타일 서비스 (MapLibre style URL)
2) MySQL 데이터베이스
3) Render (Backend 배포)
4) Vercel (Frontend 배포)

이들 서비스는 로직과 분리된 “인프라 계층”이다.

---

## C.5 환경 전환 절차(개념 단계)

OFFLINE → ONLINE 전환 시 순서:

1. Backend 배포
2. DB 환경변수 설정
3. ALLOWED_ORIGINS 설정
4. Frontend API base URL 변경
5. 지도 스타일/타일 키 확인

이 과정에서 코드 수정은 최소화되어야 한다.

---

## C.6 HTTPS 및 브라우저 보안 컨텍스트

일부 브라우저 API는 HTTPS 환경에서만 동작한다.

예:

- 화면 녹화(getDisplayMedia)
- 일부 위치 기반 API

로컬 IP(http://<ip>:port)에서는 제한될 수 있다.

배포 환경에서는 HTTPS가 기본이므로 정상 동작한다.

---

## C.7 외부 의존성과 로직의 분리 원칙

다음은 반드시 유지해야 할 설계 원칙이다.

- 지도 타일 서비스 변경은 로직에 영향을 주지 않는다.
- DB 구조 변경은 Backend 내부에서 흡수한다.
- 환경변수 변경은 코드 수정 없이 처리한다.
- Frontend는 Segment 계약만 신뢰한다.

---

## C.8 이 부록의 역할

Appendix C는 “설정 문제”를 다루는 장이다.

- 차량이 안 보일 때 → 먼저 D(운영 실패 패턴)
- 코드 수정이 필요한가? → 4장(기능 체인)
- 환경 설정 문제인가? → Appendix C

즉, 로직과 설정을 분리하는 기준점 역할을 한다.




# Appendix D — Operational Failure Patterns (운영/배포에서 자주 발생하는 실패 패턴)

본 부록은 로직 기반 매뉴얼(1–5장)과 기능 디버깅(4장)을 보완하기 위해,
**배포/환경 전환/외부 의존성**에서 반복적으로 발생하는 실패 패턴을 “증상 중심”으로 정리한다.

---

## D.1 환경 전환(OFFLINE ↔ ONLINE)에서 가장 흔한 1차 실패

### 패턴 D-1: Frontend는 뜨는데 데이터(차량)가 비어 있음
- **증상**
  - UI는 로드되지만 차량이 전혀 표시되지 않음
  - LIVE/REPLAY 모두 segments가 비어 있거나 요청 자체가 실패
- **가능 원인**
  - Frontend API base URL이 잘못된 Backend를 가리킴
  - Backend가 배포는 됐지만 DB 연결 실패(환경변수 누락)
- **확인 위치**
  - 브라우저 Network 탭에서 `/api/segments`, `/api/replay` 응답 확인
  - Backend 로그에서 DB 접속 오류 여부 확인
- **조치**
  - API base URL을 올바르게 설정
  - Backend 환경변수(DB_*) 재점검

---

### 패턴 D-2: CORS 오류로 API 호출이 막힘
- **증상**
  - 콘솔에 CORS 관련 에러
  - 브라우저가 API 응답을 차단
- **가능 원인**
  - `ALLOWED_ORIGINS` 설정 누락/부정확
- **확인 위치**
  - 브라우저 콘솔/Network 응답 헤더
  - Backend 환경변수(`ALLOWED_ORIGINS`)
- **조치**
  - 배포 URL을 허용 origin에 포함
  - 필요한 경우 와일드카드 사용(운영 정책에 따라 제한)

---

## D.2 지도(타일/스타일) 의존성 실패

### 패턴 D-3: 배경 지도만 하얗게/검게 나오거나 타일이 로드되지 않음
- **증상**
  - 지도 캔버스는 보이지만 타일이 로드되지 않음
  - “Invalid API key”, “Unauthorized” 류 메시지 가능
- **가능 원인**
  - MapLibre style URL 또는 타일 API key가 만료/오타/권한 문제 
  - style URL이 코드에 하드코딩되어 개인 계정에 종속 
- **확인 위치**
  - 브라우저 Network 탭: style.json, tile 요청 status 코드(401/403)
  - MapView.jsx의 style URL 참조 지점(환경변수화 여부)
- **조치**
  - 유효한 스타일 URL/API 키로 교체
  - 장기 운영 시 공용 계정/환경변수 주입으로 전환

---

## D.3 Render/Vercel 배포 특성(슬립/콜드스타트)로 인한 실패

### 패턴 D-4: 처음 접속할 때만 LIVE/REPLAY가 느리거나 실패함
- **증상**
  - 첫 요청이 오래 걸림(수십 초)
  - 첫 요청에서 타임아웃/502/504 이후 재시도하면 정상
- **가능 원인**
  - 배포 서버 콜드스타트/슬립 상태(특히 무료 플랜)
- **확인 위치**
  - Backend 응답 시간이 첫 요청에서만 긴지 확인
- **조치**
  - “첫 요청은 워밍업”으로 인지하고 재시도
  - 필요 시 헬스체크/워밍업 호출을 배포 후 수행

---

## D.4 REPLAY가 특정 날짜에서만 실패하는 패턴

### 패턴 D-5: 어떤 날짜는 REPLAY가 비어 있고 다른 날짜는 정상
- **증상**
  - `/api/replay?date=...`는 성공하지만 segments가 빈 배열
- **가능 원인**
  - 해당 날짜 범위에 route 데이터가 실제로 없음
  - Backend의 일 단위 조회 범위(YYYYMMDD0000~2359)와 데이터 타임스탬프가 불일치
- **확인 위치**
  - DB에서 originDeptTime 기준 해당 날짜 데이터 존재 여부
- **조치**
  - 날짜 필터 기준 컬럼(originDeptTime)과 입력 데이터 포맷 확인
  - 데이터가 존재한다면 Backend 조회 조건(경계값) 점검

---

## D.5 “코드는 같은데 동작이 다르다” 패턴(환경 차이)

### 패턴 D-6: 로컬에서는 되는데 배포에서는 안 됨
- **증상**
  - OFFLINE은 정상, ONLINE에서만 실패
- **가능 원인**
  - 환경변수 차이(DB, ORIGINS, API base)
  - 외부 의존성(Map style URL/키) 차이
- **확인 위치**
  - 1장 체크리스트(환경 전환 1차 점검)
  - 배포 환경변수 설정 화면(Render/Vercel)
- **조치**
  - “코드 수정”이 아니라 “설정 동일화”부터 수행

---

## D.6 운영 점검용 최소 체크(30초 점검)

배포 후 또는 인수인계 후, 아래 4개만 확인하면 핵심 기능은 살아있다고 볼 수 있다.

1) LIVE 모드에서 차량이 주기적으로 갱신되는가  
2) REPLAY 모드에서 특정 날짜를 선택했을 때 차량이 나타나는가  
3) areaMode를 바꿨을 때 차량/커버리지 표시가 달라지는가  
4) coverageVisible 토글이 정류장/링크 표시를 제어하는가
